KT AI/DX 직무 면접을 앞두고 연습 도구가 마땅치 않았다.  
사람인 AI 모의면접([사람인 AI 면접](https://m.saramin.co.kr/member/ai-interview)) 같은 상용 서비스는 있지만, 특정 기업의 핵심가치나 내부 평가 기준을 반영하지 않는다.  
그래서 직접 만들기로 했다. [GitHub 레포](https://github.com/dorec9/ai-interview-agent)에 전체 코드가 공개되어 있다.

---

## 1. 프로젝트 배경과 목적

목표는 단순했다. 이력서를 넣으면 "나 한테 맞는 면접 질문"이 나오고, 답변하면 항목별 피드백이 돌아오는 것.

기존 GPT 프롬프트 기반 면접 연습은 2가지 문제가 있었다.

- 질문이 맥락 없이 단발로 끊긴다. 꼬리질문이 나오지 않는다.
- 기업 고유의 평가 기준(사내 핵심가치, 예상 질문 문서)이 반영되지 않는다.

이 두 문제를 해결하는 게 설계의 출발점이었다.

---

## 2. 기술 스택과 선택 이유

| 구성요소 | 선택 기술 | 이유 |
|---|---|---|
| LLM | GPT-4o-mini | 비용 대비 충분한 추론 품질, API 안정성 |
| 에이전트 오케스트레이션 | LangGraph | 상태 기반 분기가 if/else보다 명시적 |
| 벡터 DB + RAG | Chroma + OpenAI Embeddings | 로컬 설치, 무료, 빠른 프로토타이핑 |
| 문서 파싱 | PyMuPDF, python-docx | PDF/DOCX 모두 처리 |
| UI | Gradio | 파이썬 코드 몇 줄로 웹 UI 완성 |

LangGraph를 선택한 배경을 짚어둘 필요가 있다.  
한 기술 블로그 정리에 따르면, Gartner는 2026년 기업 앱의 40%가 에이전틱 기능을 내장할 것으로 전망했다(2025년 12% 대비) ([LangGraph 2.0 정리 글](https://dev.to/richard_dillon_b9c238186e/langgraph-20-the-definitive-guide-to-building-production-grade-ai-agents-in-2026-4j2b)).  
단순 체인 대신 상태 머신 기반 에이전트 구조를 연습하기 좋은 시점이었다.

LangGraph는 그래프 구조로 복잡한 워크플로우를 표현하는 상태 기반 에이전트 오케스트레이션 프레임워크다 ([LangGraph 공식 문서](https://www.langchain.com/langgraph)).  
면접처럼 "조건에 따라 다음 질문이 달라지는" 흐름을 코드로 표현할 때 장점이 분명했다.

RAG(Retrieval-Augmented Generation)는 2026년 기준으로 실험 기술이 아니라 프로덕션 아키텍처의 일부로 자리잡았다 ([RAG Architecture Patterns](https://calmops.com/architecture/rag-architecture-retrieval-augmented-generation/)).  
KT 사내 핵심가치 문서와 예상 질문 문서를 Chroma에 넣고, 심화질문 생성 단계에서 검색하도록 연결했다.

## 3. 핵심 설계 판단

### 3-1. 면접관 3명 페르소나 분리

면접 질문을 단일 GPT 호출로 뽑으면 질문이 비슷한 패턴으로 수렴한다.  
이를 막기 위해 면접관을 3명으로 나눴다.

- **잠재력 면접관**: 성장 가능성, 학습 태도 중심
- **조직적합도 면접관**: 협업 방식, 가치관 중심
- **직무역량 면접관**: 실제 업무 능력, 전문 지식 중심

각 면접관은 경험/동기/논리 3개 영역을 담당한다. 총 9개 영역에서 질문 전략을 수립한 뒤 첫 질문을 고른다.

이 구조 덕분에 "왜 이 회사에 왔냐"는 질문과 "이 기술을 어떻게 쓸 거냐"는 질문이 섞이지 않고 각자 흐름을 탄다.

### 3-2. LangGraph 상태 머신 구조

면접 진행 에이전트는 3개 노드로 구성했다.

```
답변 입력
  → evaluate_answer (4항목 채점)
  → decide_next_step (규칙 기반 분기)
      ├─ generate_question (심화질문 + RAG)
      ├─ change_strategy (다음 영역으로 이동)
      └─ summarize_interview (보고서 생성)
```

`evaluate_answer`는 구체성, 일관성, 적합성, 논리성 4항목을 각 0/1점으로 채점한다.  
`decide_next_step`은 규칙 기반이다. 분야별 평균 점수가 75% 이상이면 다음 영역으로 넘어가고, 미달이면 심화질문을 추가로 낸다.

LLM에게 "다음에 뭘 할지" 판단을 맡기는 대신 규칙 기반 분기를 선택한 이유가 있다.  
면접 흐름은 예측 가능해야 한다. LLM이 분기를 결정하면 디버깅이 어렵고, 테스트가 어렵다.

### 3-3. 사전 준비 파이프라인 분리

면접 시작 전 파이프라인을 별도로 뺐다.

1. PDF/DOCX에서 텍스트 추출
2. GPT로 이력서 분석 (직무 키워드, 경력 사항 정리)
3. 3명 면접관 × 3개 영역 질문 전략 수립
4. 첫 질문 선택

이 과정을 인터랙티브 면접 루프와 분리하지 않으면, 답변할 때마다 이력서를 다시 읽는 낭비가 생긴다.  
사전 준비를 한 번만 실행하고 상태로 넘기는 구조가 비용과 응답 속도 양쪽에서 유리했다.

## 4. 배운 점과 개선할 점

### 배운 것

**LangGraph의 상태 관리는 생각보다 명시적이어야 한다.**  
초기에 상태 스키마를 느슨하게 정의했다가 노드 사이에서 데이터가 유실되는 문제를 겪었다.  
`TypedDict`로 상태를 엄격하게 정의하고 나서야 디버깅이 수월해졌다.

**평가 기준의 단순화가 일관성을 높인다.**  
4항목 0/1점 채점은 단순해 보이지만 GPT가 매번 다른 채점을 내는 문제를 줄였다.  
루브릭이 명확할수록 LLM 출력의 분산이 줄어든다는 걸 수치로 확인했다.

**RAG 문서 품질이 결과 품질을 결정한다.**  
Chroma에 넣은 문서가 정리되지 않았을 때는 심화질문 품질이 낮았다.  
청크 크기(chunk size)와 문서 정제 수준이 검색 품질에 직접 영향을 준다.

### 개선할 것

**음성 입력 지원.**  
실제 면접은 말로 하는데, 지금은 텍스트 입력만 가능하다.  
Whisper API를 Gradio에 연결하면 실제 면접 환경에 가까워진다.

**평가 기준의 세분화.**  
0/1점 채점은 피드백의 해상도가 낮다.  
0~3점 척도로 바꾸고 세부 기준을 추가하면 더 유의미한 피드백이 나온다.

**기업별 문서 교체 인터페이스.**  
지금은 KT 타깃 문서가 하드코딩되어 있다.  
UI에서 문서를 교체할 수 있게 만들면 다른 기업 면접 준비에도 쓸 수 있다.

---

이 프로젝트를 만들면서 LangGraph가 단순 체인보다 복잡한 이유를 몸으로 알게 됐다.  
상태를 명시적으로 관리하고, 분기 조건을 코드로 표현하는 습관이 에이전트 설계의 기본이다.  
다음 버전에서는 음성 입력과 다기업 지원을 붙여볼 계획이다.
