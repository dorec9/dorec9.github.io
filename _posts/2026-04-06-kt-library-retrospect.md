2025년 12월, KT AivleSchool 과정의 팀 프로젝트로 도서 관리 웹 서비스를 만들었다. 12/9부터 12/12까지 4일, 30개 이상의 커밋이 쌓였다. 짧은 기간에 어떤 판단을 했고, 어디서 막혔는지 기록한다.

---

## 프로젝트 배경

서비스 핵심은 단순하다. 사용자가 텍스트(제목·내용)를 입력하면 AI가 표지 이미지를 생성하고, 다른 사용자의 작품도 열람할 수 있는 커뮤니티형 도서 플랫폼이다. KT AivleSchool 9조가 만들었고, 소스코드는 [dorec9/kt_library](https://github.com/dorec9/kt_library)에 있다.

기능 범위는 도서 CRUD, AI 표지 생성, 좋아요·찜·댓글, 사용자 인증, 인기순·최신순 정렬, 마이페이지로 잡았다.

---

## 기술 스택 선택 이유

**백엔드: Java 17 + Spring Boot 3.x + H2**

Spring Boot를 선택한 이유는 팀 내 Java 사용 경험이 가장 많았기 때문이다. Spring Boot 3.x는 Java 17 이상을 요구하므로 버전도 17로 고정했다.

DB는 H2를 썼다. 4일 프로젝트에서 별도 DB 서버를 구성하는 비용이 납득되지 않았다. 인메모리 모드로 띄우면 스키마 초기화가 자동으로 되고, 팀원이 로컬에서 바로 실행할 수 있다. 프로덕션 전환 없이 구조를 검증하는 용도로는 적합했다.

**프론트엔드: React (Vite) + JavaScript**

페이지는 Home, BookList, BookDetail, BookCreate, BookEdit, Login, Signup, MyPage 8개다. React의 컴포넌트 단위 분리가 이 정도 규모에서 페이지 간 상태 공유를 관리하기에 맞았다. Vite는 번들 속도보다 개발 서버 HMR 속도 때문에 선택했다.

**ORM: Spring Data JPA + Lombok + MapStruct**

JPA로 도메인 6개(Book, Comment, Favorite, Image, Like, User)를 매핑했다. Lombok으로 보일러플레이트를 줄이고, MapStruct로 엔티티↔DTO 변환 코드를 생성했다. 수동으로 변환 메서드를 작성하면 DTO가 늘어날수록 실수가 생긴다. 컴파일 타임에 생성되는 MapStruct가 그 위험을 줄였다.

## 핵심 설계 판단 3가지

**1. Service 인터페이스 + 구현 분리**

`service/` 하위에 인터페이스를 두고 `service/impl/`에 구현체를 분리했다. 4일 프로젝트에서 구현체가 교체될 가능성은 낮다. 그럼에도 이 구조를 택한 이유는 테스트 작성 시 Mock 주입이 편하고, 컨트롤러가 구현 세부사항에 의존하지 않는다는 점에서다. 단기 프로젝트라도 레이어 경계를 지키는 습관이 중요하다고 판단했다.

**2. DTO 요청·응답 분리**

`dto/request`와 `dto/response`를 디렉터리 수준에서 나눴다. 요청 DTO에는 유효성 검증 어노테이션이 붙고, 응답 DTO에는 클라이언트에 노출할 필드만 담긴다. 엔티티를 그대로 내보내면 불필요한 필드 노출, 순환참조 직렬화 오류가 생긴다. 이 분리는 초기 비용이 있지만 디버깅 비용이 더 크다고 봤다.

**3. Global Exception Handler + 커스텀 예외**

`@ControllerAdvice`로 GlobalExceptionHandler를 만들고, 도메인별 커스텀 예외 클래스를 정의했다. 컨트롤러마다 try-catch를 반복하지 않아도 되고, 예외 메시지를 한 곳에서 포맷화할 수 있다. AOP 기반이므로 핸들러를 추가해도 비즈니스 로직이 오염되지 않는다.

**인증: 세션 기반 + Spring Interceptor**

JWT 대신 세션을 선택했다. 4일 안에 토큰 갱신 로직, 클라이언트 토큰 저장 전략까지 구현하는 것은 무리라고 판단했다. LoginCheckInterceptor를 등록해 인증이 필요한 URL 패턴만 세션 검사를 통과하도록 했다. 자동 로그아웃도 세션 만료 시간 설정으로 처리했다.

## 개발 과정에서 부딪힌 문제

**좋아요 토글 3회 수정**

커밋 이력을 보면 좋아요 기능이 ver.1 → ver.2 → ver.3 순서로 3번 수정됐다. 원인을 정확히 특정하기 어렵지만, 커밋 패턴으로 보면 프론트-백 상태 동기화 문제였을 가능성이 높다. React에서 토글 후 서버 응답이 오기 전에 로컬 상태를 갱신하면, 실제 DB 상태와 UI가 어긋난다. 특히 낙관적 업데이트(Optimistic Update)와 서버 응답을 맞추지 않으면 버튼이 예상과 반대로 동작하는 증상이 나온다.

3번의 수정을 거쳤다는 것은 처음에 이 동기화 전략을 명확히 정의하지 않았다는 뜻이다.

**AI 이미지 생성 API 교체**

OpenAI의 DALL-E API로 표지 이미지를 생성하는 기능을 먼저 구현했다. 이후 비용 문제로 무료 API로 전환 테스트를 진행했다. 4일 프로젝트에서 API 비용은 실제 제약이다. 인터페이스를 추상화해두지 않으면 API 교체 시 서비스 코드 전체를 수정해야 한다.

## 배운 점과 개선할 점

**배운 점**

첫째, 레이어 경계를 명확히 하면 수정 범위가 작아진다. DTO 분리와 Service 인터페이스 구조 덕분에 기능 추가 시 다른 레이어를 건드리는 경우가 줄었다.

둘째, 상태 변경 기능은 동기화 전략을 먼저 정해야 한다. 좋아요 토글처럼 즉각 피드백이 필요한 기능은 낙관적 업데이트 여부와 에러 롤백 방식을 설계 단계에서 결정해야 한다.

셋째, 외부 API 의존 기능은 교체 가능성을 열어두어야 한다. AI 이미지 생성처럼 비용 변수가 있는 기능은 인터페이스 뒤로 숨겨 구현체를 쉽게 교체할 수 있게 해야 한다.

**개선할 점**

H2는 인메모리 모드에서 서버 재시작 시 데이터가 사라진다. 파일 모드로 전환하거나 초기 데이터 SQL을 관리했으면 데모 준비가 수월했을 것이다. 좋아요·댓글처럼 실시간성이 있는 기능은 프론트와 백이 API 명세를 먼저 맞추고 개발을 분리했으면 수정 횟수를 줄일 수 있었다.

---

4일 안에 백엔드·프론트·AI 기능을 모두 붙이는 것은 쉽지 않았다. 커밋 이력이 그 과정을 그대로 보여준다. 다음 팀 프로젝트에서는 API 명세 합의와 외부 의존 기능의 교체 전략을 초반에 잡는 것을 우선순위로 두겠다.
